# ERP 物料主数据治理 Agent PRD

## Problem Statement

ERP 物料主数据中存在名称不同但实际为同一物料的情况，例如“西红柿”和“番茄”，容易形成一物多码。申请人通常不知道物料是否已存在，也无法提前找到可复用的物料编码，导致重复申请、名称和规格录入不规范，以及后续主数据维护成本上升。

核心使用通用物料模型；当前 NCC Adapter 从 SQL Server 视图 `bd_material_v` 映射 `code`、`name`、`materialspec`、`materialshortname`、`ename`、`memo`、`enablestate` 等厂商字段。`materialtype` 存在大量空值，不作为核心判定条件；`code` 只用于定位和展示。

## Solution

建设一个内网可复用的、只读 API-first 物料主数据 Agent。它通过 `MaterialSource` Adapter 从 ERP 读取通用物料数据，在 PostgreSQL 中建立标准化文本和向量检索索引，通过确定性规则、关键词检索、向量召回和本地模型精排，为其他系统提供：

1. 按名称或规格型号查询已有物料编码。
2. 新建物料前判断是否已有相同或疑似重复物料。
3. 提示名称、规格型号和单位的企业录入建议。
4. 接收物料主数据管理员的确认结果，积累同义词和反例。

Agent 只提供提醒、推荐和证据，不直接创建、修改、启用、停用或合并 ERP 物料，也不自动生成物料编码。

## User Stories

1. 作为物料申请人，我希望按物料名称查询已有物料，以便找到可复用的物料编码。
2. 作为物料申请人，我希望按规格型号查询物料，以便在不知道名称时仍能定位已有物料。
3. 作为物料申请人，我希望同时输入名称和规格型号进行查询，以便获得更准确的结果。
4. 作为物料申请人，我希望输入“番茄”时能够找到名称为“西红柿”的物料，以便识别常见同义词。
5. 作为物料申请人，我希望系统能识别大小写、全半角、空格、标点和常见单位写法差异，以便减少格式差异造成的漏检。
6. 作为物料申请人，我希望新建前得到“已有可用物料”的明确提示，以便优先使用现有编码。
7. 作为物料申请人，我希望新建前得到“已有待启用物料”的提示，以便联系管理员评估是否复用。
8. 作为物料申请人，我希望命中停用物料时看到停用状态，以便避免误把历史物料当作当前可用物料。
9. 作为物料申请人，我希望系统在无法确定时显示“疑似重复，人工确认”，以便我不会把模型建议误认为确定结论。
10. 作为物料申请人，我希望系统在未发现重复时明确提示可以申请新建，以便继续办理业务。
11. 作为物料申请人，我希望系统指出名称或规格缺失的关键要素，以便一次性补齐申请信息。
12. 作为物料申请人，我希望系统提示不规范的名称、规格和单位写法，以便提高物料录入质量。
13. 作为物料主数据管理员，我希望看到候选物料的 `code`、`name`、`specification` 和 `status`，以便核对建议。
14. 作为物料主数据管理员，我希望看到系统为何判断两个物料相似，包括命中的同义词、字段和规格冲突，以便快速审核。
15. 作为物料主数据管理员，我希望确认“同物”或“不同物”，以便沉淀可复用的判重样本。
16. 作为物料主数据管理员，我希望记录“西红柿=番茄”这样的同义词，以便后续查询能够稳定命中。
17. 作为物料主数据管理员，我希望记录“圣女果≠普通番茄”这样的反例，以便降低错误合并风险。
18. 作为物料主数据管理员，我希望查看规则版本、索引版本、更新时间和覆盖记录数，以便判断结果是否新鲜。
19. 作为物料主数据管理员，我希望手动触发索引重建，以便在 ERP 数据变更后及时更新检索能力。
20. 作为集成系统，我希望通过同步 JSON API 查询物料，以便在 3～5 秒内展示查重结果。
21. 作为集成系统，我希望通过独立的查询接口和校验接口获得稳定语义，以便分别支持“找编码”和“新建前校验”。
22. 作为集成系统，我希望查询结果包含置信度等级和解释，以便决定是否展示提醒或转人工审核。
23. 作为系统管理员，我希望 Agent 通过 Adapter 只读访问 ERP，以便不破坏正式数据。
24. 作为系统管理员，我希望 Agent 使用 PostgreSQL 保存检索索引，而不是修改 ERP 表，以便隔离检索性能和主数据风险。
25. 作为系统管理员，我希望能查看索引版本并手动同步，以便减少索引延迟造成的漏检。
26. 作为系统管理员，我希望所有结果可追溯到索引版本、规则版本和管理员反馈，以便排查误判。
27. 作为业务负责人，我希望第一版覆盖全部物料，而不是只绑定某一个物料类别，以便统一接入不同系统。
28. 作为业务负责人，我希望没有正式依据的内容只称为“企业录入建议”，以便避免把模型猜测冒充行业标准。
29. 作为业务负责人，我希望 Agent 完全只读，以便把最终创建、修改、启用、停用和合并责任保留给现有 ERP 流程。
30. 作为业务负责人，我希望后续可以复用同一套 API 做存量治理，以便在新建防重稳定后逐步清理历史数据。

## Implementation Decisions

- **数据源与事实来源**：配置的 ERP 是正式物料数据的唯一事实来源，核心通过只读 `MaterialSource` interface 访问。当前内置 NCC Adapter，其 `bd_material_v` 字段映射不进入核心层。
- **索引存储**：使用官方 `langchain-postgres` 的 `PGVector`，collection 名为 `erp_materials`；物料保存为 LangChain `Document`，通用字段保存在 JSONB metadata 中，物料编码作为文档 ID 进行幂等更新。
- **模型分工**：Embedding 模型负责从十万级数据中召回约 20～50 个候选；本地对话模型只处理候选精排、结论解释和录入建议；精确匹配、格式标准化和规格冲突优先使用确定性规则。
- **匹配字段**：`name` 是主判定字段，`specification` 是关键规格字段；`short_name` 和 `english_name` 只作辅助召回；`description` 默认只展示；`code` 只返回、不参与相似度计算。
- **标准化**：处理大小写、全半角、空格、标点、常见单位别名和明显数值单位换算；无法安全解析的规格保留原文并降低置信度。
- **语义判定**：名称和别名负责召回候选，关键规格属性负责避免误合并；“西红柿/番茄”可作为同义词命中，“圣女果/普通番茄”必须允许作为反例。
- **空值策略**：`name` 和 `specification` 至少填写一个；只填名称或只填规格允许查询，但不能轻易给出确定重复；两者均为空直接拒绝请求。
- **状态策略**：核心状态为 `inactive`、`active`、`disabled`；具体 ERP 状态由 Adapter 映射。停用物料继续参与查重，但不自动建议恢复。
- **固定结论**：已有可用物料、已有待启用物料、疑似重复人工确认、未发现重复可申请新建。
- **API**：提供 `search` 查询已有物料、`validate` 校验拟新建物料、`feedback` 接收管理员确认、`index/rebuild` 手动重建索引、`index/status` 查看索引状态。第一版只支持单条同步查询，不支持批量接口。
- **索引触发**：索引重建通过 `POST /api/v1/materials/index/rebuild` 手动触发，后台异步执行；通过 `GET /api/v1/materials/index/status` 查询任务状态、同步数量和错误信息，不依赖命令行操作。
- **自然语言入口**：对外主入口为 `POST /api/v1/agent/chat`，请求只需提供自然语言 `message`；LangChain 1.x `create_agent` 根据意图调用 `search_materials` 或 `validate_new_material` 工具，再以自然语言返回结论和证据。固定字段接口保留为底层服务 seam 和给非 Agent 系统的可选能力，不作为主要交互方式。
- **性能目标**：`search` 和 `validate` 在正常索引状态下 3～5 秒内返回；索引任务异步执行。返回索引版本，不把索引结果伪装成 ERP 实时查询。
- **录入建议**：没有既有行业或企业规范时，只根据现有数据统计和管理员反馈形成“企业录入建议”，不得声称符合行业标准；规则必须版本化、可追溯、可撤销。
- **权限与写入边界**：第一版限定内网调用，Agent 对 ERP 完全只读，不创建、修改、启用、停用、合并物料，也不生成物料编码。
- **反馈学习**：管理员确认结果才可进入同义词和反例库，记录确认人、时间、规则版本和来源；反馈不直接回写 ERP。
- **索引运维**：支持手动重建；显示索引版本、最后更新时间、覆盖记录数和失败记录数；索引只是缓存，不能替代 ERP 事实数据。
- **项目 seam**：以 API 层作为最高测试 seam，验证请求、响应、结论、候选证据和错误处理，不绑定具体模型、向量库内部实现。
- **Agent 框架**：采用 LangChain 1.x 作为模型、工具和结构化输出层，采用 LangGraph 1.x 编排可审计的判定流程；不使用一个无限循环的万能 Agent 承担全部职责。
- **当前环境基线**：开发环境已安装 `langchain 1.3.12`、`langgraph 1.2.8`、`langchain-core 1.4.9` 和 `langchain-openai 0.3.22`。正式项目应以开发时验证过的最新稳定版本为准，并通过锁文件固定版本，不能使用不受控的 `latest` 或 `*`。
- **模型接入**：本地模型若提供 OpenAI-compatible API，使用 LangChain 的 `ChatOpenAI`/`init_chat_model` 适配；Embedding 使用兼容的 Embeddings 实现。模型地址、模型名和密钥通过环境变量注入，不写入代码。
- **工作流节点**：LangGraph 状态图按顺序编排“输入校验 → 文本标准化 → 精确/关键词检索 → 向量召回 → 规格规则校验 → 本地模型精排 → 结论和录入建议 → 审计记录”，每个节点输入输出可测试、可重试、可追踪。
- **结构化结果**：使用 Pydantic schema 约束 API 和模型输出，固定结论、置信度等级、候选物料、匹配依据、冲突字段、录入建议和版本信息，禁止依赖自由文本解析。
- **工具边界**：LangChain tools 只暴露只读物料查询、索引状态查询和反馈写入 Agent 自有库的能力；不提供 ERP 写入工具。新建物料、启停用、合并等动作不进入 Agent 工具集合。
- **依赖取舍**：新项目不引入 `deepagents` 作为基础依赖。当前需求是有限步骤的确定性工作流，LangGraph 已足够；只有出现明确的长任务、人工中断恢复或多代理协作需求时再增加抽象。

## Testing Decisions

- 测试只验证外部行为，不测试模型内部实现或具体向量距离数值。
- API 合同测试：验证名称查询、规格查询、名称加规格查询、空输入拒绝、四类固定结论和响应字段完整性。
- 判重行为测试：验证精确匹配、同义词匹配、规格单位标准化、规格冲突降级、停用物料参与查重和状态展示。
- 录入建议测试：验证缺失字段、不规范单位、名称要素不足和无规则时的“企业录入建议”措辞。
- 索引测试：验证手动重建、索引状态、版本变化、失败记录和索引过期提示。
- 反馈测试：验证管理员确认后同义词和反例可被后续查询使用，并能撤销。
- Adapter 合同测试：验证厂商字段、启用状态和修改时间能稳定映射到通用物料模型。
- 性能测试：以十万条以上物料规模验证单条 `search` 和 `validate` 的 3～5 秒目标。
- 当前仓库没有可复用的 ERP 物料测试先例，因此第一版应先建立 API 合同和端到端测试 seam。
- LangChain/LangGraph 测试使用假模型、假 Embeddings 和固定检索夹具，验证状态图的外部结果，不调用真实本地模型完成单元测试。
- 不建立独立的大规模人工标注集；通过管理员反馈持续积累样本，并在上线后记录召回、误报和人工确认耗时。

## Out of Scope

- 自动创建、修改、启用、停用或合并 ERP 物料。
- 自动生成物料编码。
- 自动批准或阻止新建申请。
- 批量存量物料清理和自动合并。
- 多租户、互联网开放访问和复杂外部权限体系。
- 把模型生成内容直接认定为国家标准、行业标准或企业制度。
- 一开始为所有物料类别建立复杂的行业专用属性解析器。
- 用 `materialtype` 作为必需判定字段。
- 将 `memo` 作为主要判重依据。

## Further Notes

- 第一阶段应先完成只读索引、`search`、`validate` 和管理员反馈闭环，再评估存量清理。
- 误判策略偏向“宁可多提示，不漏掉明显重复”，但中等置信度只能提醒，不能自动阻止业务。
- “已有可用物料”应优先建议复用编码；“已有待启用物料”应提示联系管理员；“已停用物料”只作为风险证据展示。
- ERP 数据变更后可手动重建索引；索引结果必须展示更新时间，避免调用方误以为是实时全库结果。
- PostgreSQL 索引库与 ERP 正式库分离，所有索引和反馈写入都不扩大 ERP 写入风险。
